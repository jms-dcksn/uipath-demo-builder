# Demo Builder — Skill Improvements Plan

Action plan for fixes surfaced by the Acme Bank Loan Origination build (2026-04-24). Source: post-build review of `caseplan.json` deployability against UiPath Studio Web requirements.

All paths below are inside the installed plugin cache:
`/home/jamesdickson/.claude/plugins/cache/uipath-demo-builder/demo-builder/0.1.0/`

The canonical edits should land in the source repo (`uipath-demo-builder` plugin) and be re-released. Do NOT hand-edit the cache — changes will be overwritten on next plugin update.

## Priority 0 — Deployment-blocker fixes (do first)

### P0-1. Remove the caseplan.json hand-synthesis fallback in `case-designer`

**File:** `agents/case-designer.md`

**Problem:** Sub-agent has no `Skill` tool. When step 7 says "delegate to the installed `uipath-case-management` skill", it can only look for a sibling file, doesn't find it (the skill lives in a different plugin), and silently hand-synthesises a custom JSON. The output looks like a caseplan but is not the UiPath schema and cannot be deployed.

**Fix:**
- Drop step 7 ("Delegate synthesis of `tasks.md` and `caseplan.json`").
- End the sub-agent's responsibility at `sdd.md` (steps 3-6 stay).
- Add a hard rule under "How to work": *"You MUST NOT author `caseplan.json` directly. If `sdd.md` cannot be completed, return an error to the architect — do not produce any file at `flow-model/caseplan.json`."*
- Update the "Output to the architect" block: remove the `caseplan.json` location and generator-warnings bullets; add a single bullet — *"`sdd.md` ready for delegation to `uipath-case-management`."*

### P0-2. Move `uipath-case-management` delegation to the planner thread

**File:** `skills/demo-builder-planner/SKILL.md` and `skills/demo-builder-planner/references/delivery-workflow.md`

**Problem:** Only the architect (planner, main thread) has the `Skill` tool. The sub-agent cannot invoke a procedural skill.

**Fix:** In the Phase 3 description (both files), explicitly split into two steps:

```
Phase 3a — Case Management design (sub-agent)
  Dispatch case-designer → returns sdd.md and case-management-design.md.

Phase 3b — caseplan.json generation (architect, main thread)
  Architect invokes uipath-case-management skill via the Skill tool, passing
  builds/<demo-slug>/flow-model/sdd.md as input. Skill produces tasks.md
  (requires user approval), then caseplan.json.
```

Update the dispatch table — Phase 3 row becomes two rows.

### P0-3. Update `demo-builder-case-management` skill to match

**File:** `skills/demo-builder-case-management/SKILL.md`

**Problem:** Step 4 ("Delegate to the installed `uipath-case-management` skill...") describes work the sub-agent cannot perform.

**Fix:**
- Replace step 4: *"Return `sdd.md` to the architect. The architect (planner thread) owns delegation to `uipath-case-management`. Do not invoke that skill from inside this skill."*
- Adjust Completion Criteria — remove "tasks.md and caseplan.json generated by the delegated skill"; add "sdd.md handed back to architect for delegation."

## Priority 1 — sdd.md completeness (do second)

### P1-1. Extend `sdd.template.md` with the four missing sections

**File:** `skills/demo-builder-case-management/templates/sdd.template.md`

**Problem:** Today's sdd.md has stages, tasks, and exception paths but no **Trigger Type**, no **SLA**, no formal **Case Exit Conditions**, and no **Variables / Arguments** section. The production `uipath-case-management` skill plans from sdd.md — anything not declared there will not appear in `caseplan.json`.

**New sections (added to template, not to live builds — those re-derive from template):**

```markdown
## Trigger Type
One of: manual / timer / event. If event, name the IS connector + payload schema.
Justify the choice.

## Variables / Arguments
| Var ID | Scope (input/output/inputOutput) | Type | Source field | Notes |
|---|---|---|---|---|

Default: case entity fields are surfaced via =datafabric.<Entity>.<field>.
List any *additional* global variables not on the entity here.

## SLA
**Default case SLA:** <e.g., 30 days from CASE_OPENED to FUNDED>
**Conditional SLA rules:**
| Rule ID | Trigger | Deadline | Escalation |
|---|---|---|---|

At minimum declare one rule per regulatory deadline (e.g., TRID LE 3-day).

## Case Exit Conditions / Terminal States
| Terminal State | Status | currentStage | Trigger condition | Notes |
|---|---|---|---|---|

One row per terminal state. Each becomes a case-exit-condition in the caseplan.
```

### P1-2. Add a stub-contract → SDD threading rule

**File:** `skills/demo-builder-case-management/references/stub-contract-rules.md`

**Fix:** Add a closing note: *"Every stub output that is referenced in a case-exit-condition, SLA rule, or trigger payload MUST also appear in the corresponding §SDD section. The case-management-design.md is the catalog; sdd.md is the contract for the production skill."*

## Priority 2 — Process gaps in the planner (do third)

### P2-1. Phase 0 preflight — CLI capability check

**File:** `skills/demo-builder-planner/SKILL.md`

**Problem:** A missing `uip maestro case` surface (or `uip codedagent`, or `uip solution`) is only discovered mid-build, after expensive sub-agent dispatches.

**Fix:** Add a Phase 0 step before discovery:

```markdown
## Phase 0 — Preflight (architect, < 30s)

Run and record on `manifest.md`:
- `uip --version`
- `uip login status --output json`
- `uip maestro case --help` (must list registry/cases/stages/tasks/edges/var/sla/validate)
- `uip codedagent --help`
- `uip solution --help`

If any required surface is absent, abort the build and surface a single
blocker message to the user. Do NOT proceed to discovery. The user must
upgrade the CLI or switch machines first. Record CLI version and login
state on manifest.md regardless of outcome.
```

### P2-2. Phase 5 — strict parallel dispatch rule for AG-*

**File:** `skills/demo-builder-planner/SKILL.md` and `skills/demo-builder-planner/references/delivery-workflow.md`

**Problem:** The "single-turn parallel" rule is in prose but easy to skim past; AG-01 and AG-02 were dispatched sequentially in the Acme build.

**Fix:** Promote to a checklist:

```markdown
## Phase 5 — Agents (HARD CHECKLIST)

Before dispatching:
- [ ] Enumerate every AG-* ID from the task matrix.
- [ ] Confirm each has a 1:1 mapping to a planned project directory.

Dispatch:
- [ ] Send ONE assistant message containing N parallel `Agent` tool calls
      (one per AG-*). Sequential dispatch is a build error.
- [ ] If you find yourself about to send a second turn before all AG-* are
      dispatched: STOP. Re-issue as a single parallel turn.

After dispatch:
- [ ] Each report read once. Mismatches between agent outputs and
      example records → Phase 5.5 fixture consistency check.
```

### P2-3. New Phase 5.5 — Fixture consistency check

**File:** `skills/demo-builder-planner/SKILL.md` and `references/delivery-workflow.md`

**Problem:** AG-02 deterministically produces `APPROVE_WITH_CONDITIONS` for the URGENT example ($1.48M loan + R-HIGH-VALUE-REVIEW), but the example record was authored as `APPROVE`. Inconsistencies like this were caught only post-hoc by the agent-builder's own report.

**Fix:** New phase between 5 and 6:

```markdown
## Phase 5.5 — Fixture consistency check (architect)

For each example record in case-entity/case-entity.example.json:
1. Compute the deterministic output of every AG-* against that record's
   inputs (read each agent's policy catalog or fixture map).
2. Compare against the record's persisted agent-output fields
   (`decision.*`, `conditions[]`, `income.monthlyVerifiedUSD`, etc.).
3. On mismatch: patch the example record to match the agent OR escalate
   to user if the agent contract is wrong.

Output: a one-page consistency report in case-entity/fixture-consistency.md
listing every checked field and resolution. Required before Phase 6.
```

### P2-4. New Phase 6.5 — Frontend ↔ schema reconcile

**File:** `skills/demo-builder-planner/SKILL.md` and `references/delivery-workflow.md` and `agents/frontend-builder.md`

**Problem:** Frontend-builder added three fields (`decision.policyCitations[]`, `decision.rationale`, `decision.reservesMonths`) directly to fixtures without going through the schema. Caseplan/Data Fabric will not have those fields.

**Fix two ways (do both):**

1. Hard rule in `agents/frontend-builder.md`:
   *"You MUST NOT add fields to fixtures that are not present in `case-entity/case-entity.schema.json`. If a field is needed, return to the architect with a list of additions; do not modify fixtures inline."*
2. New Phase 6.5 in planner:
   ```
   After frontend-builder reports, diff the TypeScript types in
   frontend/src/types/ against case-entity.schema.json. Any field in
   fixtures.ts not present in the schema → re-dispatch data-modeler in
   reconciliation mode to add. Required before Phase 7.
   ```

### P2-5. Manifest auto-update at every phase boundary

**File:** `skills/demo-builder-planner/SKILL.md`

**Problem:** `manifest.md` only got populated at end of build, by hand. Mid-build state was invisible.

**Fix:** Add to each Phase block in the planner:

```markdown
Exit step (every phase): append to manifest.md the phase status
(DONE/SKIPPED/FAILED), artifact paths produced, and any open issues.
Use the Edit tool, never overwrite — multiple writers preserve order.
```

### P2-6. Single source of truth for cross-sub-agent questions

**File:** `agents/case-designer.md`, `agents/data-modeler.md`, `agents/agent-builder.md`, `agents/frontend-builder.md`

**Problem:** The data-modeler returned 6 open questions; case-designer answered 4 of them in its own report; nothing updated the data-modeler's notes file. Future readers see stale "open" questions in `data-fabric-modeling-notes.md`.

**Fix:** Add to each sub-agent's instruction set:

> When you resolve a question raised by a previous sub-agent's notes file,
> Edit that notes file and mark the question RESOLVED with your resolution
> inline. Do not let questions go stale across sub-agent boundaries.
